# Game Deck Cover Library Design

Date: 2026-07-13

Status: approved production direction

## Goal

Give all twenty-two playable Game Deck cartridges original, lore-native cover
art and make that art the visible front door to each game in the shared
library. The set must look like one collectible product family while preserving
the two fictional hardware families and every game's declared display aspect.

The deliverable is complete only when every catalog row has a final cover,
every cover is visible in the real keyboard/controller-accessible library, and
one contact sheet lets the owner review the complete collection at a glance.

## Chosen approach

Use a unified collectible line built from twenty-two separately generated,
text-free key-art illustrations and one deterministic packaging compositor.
The compositor adds exact titles, hardware branding, aspect and player badges,
consistent borders, and subtle printed wear. This keeps the illustration
expressive while preventing misspelled AI lettering and inconsistent product
marks.

Two alternatives were rejected:

1. Baking heavy scratches, tape, and sun damage into every illustration would
   strengthen the artifact feeling but reduce title and thumbnail legibility.
2. Pristine pre-collapse retail boxes would be readable but would not feel
   naturally handled, traded, and rediscovered inside DRIVN.

The accepted balance is clean master art with restrained edge rub, ink noise,
and dust in the packaging pass. Physical world objects may add stronger wear
later through their materials without damaging the source covers.

## Product families

### Safehouse Console

All twelve console games use a 1024 x 1536, 2:3 portrait box front. A carbon
black upper rail reads `SAFEHOUSE CONSOLE`; an amber `16:9` badge sits opposite
it. The lower title block uses bone lettering, one title-specific accent, and a
small `1-N PLAYERS // LOCAL // ONLINE` capability line derived from the catalog.
The cover is a box front for a game that renders on the console's shared 16:9
display; the package itself does not become 16:9.

### Pocket Game Deck

The ten handheld games use cartridge-label masters whose canvas matches the
screen format the cartridge was designed for:

- 1:1: 1024 x 1024;
- 9:16: 864 x 1536; and
- 16:9: 1536 x 864.

A narrow top rail reads `POCKET GAME DECK`, with the exact aspect badge on the
opposite side. The title sits in a protected lower band sized for the format.
The shell presents every label inside a common library card without stretching
or cropping it.

Both families use the established Game Deck ink, bone, amber, rust, and
signal-teal language. Individual covers may add red, oxide orange, acid green,
sun-faded blue, or emergency yellow. Purple is forbidden.

## Shared art rules

- Every illustration is original DRIVN material generated for this feature.
- No real console mark, publisher logo, certification seal, trademarked
  character, upstream game art, upstream map, or copied package layout appears.
- The generated key art contains no lettering, numbers, logos, UI, border,
  watermark, or signature. Exact text is added only by the compositor.
- Composition must remain readable at a 180-pixel-wide library thumbnail.
- Each image has one dominant silhouette, one secondary gameplay cue, and
  deliberate negative space for the packaging rails.
- Visual language is screen-printed pulp science fiction: bold shapes,
  halftone texture, distressed ink, practical machinery, and late-1980s to
  late-1990s illustrated game-box energy. It must not imitate a named artist.
- People, uniforms, vehicles, arenas, and insignia are original to DRIVN.
- Wear remains subtle: light edge rub, misregistration, dust specks, and faded
  ink only. No title may become difficult to read.
- Key art is center-safe so the deterministic crop can produce the required
  format without cutting off the dominant subject.

## Generation and composition pipeline

Use the built-in image generation tool once per game with the shared rules and
that game's brief. Each prompt uses the `stylized-concept` taxonomy and names
the intended canvas orientation. Generated files are temporary key art until
they pass subject, originality, palette, composition, text-free, and
watermark-free inspection.

A repository tool, `tools/game_deck/build_covers.py`, performs only
deterministic packaging work:

1. center-crop and resize approved key art to the required final dimensions;
2. add the family rail, protected title band, border, and exact catalog title;
3. add aspect, player-count, local/online, and cartridge-format badges from a
   checked-in cover manifest;
4. apply one seeded, restrained print-wear layer; and
5. write high-quality WebP files to
   `game/assets/game_covers/<game_id>.webp`.

The tool uses a redistributable font checked into
`game/assets/fonts/game_deck/` so output does not depend on a workstation font.
The font's license text is stored beside it. Temporary generated key art stays
outside `res://` and is removed after accepted finals are built; the full prompt
set and art briefs remain in this document for reproducibility.

The final contact sheet is written to
`docs/verification/GAME_DECK_COVERS_CONTACT_SHEET.webp`. It has separate
POCKET GAME DECK and SAFEHOUSE CONSOLE sections, preserves each cover's aspect,
and adds no review-only alterations to the individual deliverables.

## Catalog contract

Every row in `game/data/games.json` receives one field:

```json
"cover_path": "res://assets/game_covers/waste_heap.webp"
```

`cover_path` must be non-empty, unique, loadable as `Texture2D`, and point to a
file whose pixel aspect matches the row's packaging law. The existing `aspect`
continues to describe the game screen. Console cover validation expects 2:3;
handheld cover validation expects the row's 1:1, 9:16, or 16:9 value.

Missing or invalid art must not prevent DRIVN from booting. The library uses a
deterministic ink-and-amber fallback card and logs the bad catalog path. The
focused catalog test fails so the defect cannot ship unnoticed.

## Library experience

Replace the text-only vertical cartridge list with a scrollable grid of
focusable cover cards. A focused card shows the complete cover without crop,
the existing title/aspect/power/network line, and its owned state. Locked cards
keep their cover visible under a dark wash and display `LOCKED - FIND
CARTRIDGE`; unavailable development rows display `NOT INSTALLED`.

`ProtoGameCoverCard` is a focused `Button` component. It owns the cover image,
badges, lock overlay, hover/focus treatment, and accessible text. The shell
owns filtering, ordering, launch/lobby routing, and focus restoration exactly
as it does now. The card's `text` property retains the current searchable
library string so existing accessibility helpers and tests do not lose their
semantic surface.

The grid uses four columns at the 1280 x 720 console target and two columns at
the shell's 720 x 600 minimum. Handheld labels use `STRETCH_KEEP_ASPECT_CENTERED`
inside a common art well; square, portrait, and landscape art is never
stretched. Keyboard arrows, mouse, and controller focus all reach every card.
Hover, focus, pressed, disabled, owned, and locked states remain visually
distinct without relying on color alone.

Selecting a console card continues to open MATCH. Selecting a handheld card
continues to open PLAY. Covers do not alter ownership, power, networking,
scores, saves, game rules, or cartridge lifecycles.

## Attribution and provenance

The fronts contain no third-party marks. Existing code and source attribution
remains in the aggregate notices and in-game ABOUT surfaces. Generated cover
art is not presented as upstream project artwork and does not change the
license classification of any game implementation.

The checked-in compositor font must permit commercial redistribution and
modification. Its license notice ships beside the font and is added to the
aggregate notice if that license requires attribution.

## Testing contract

Create `game_cover_sim.gd/.tscn` and prove through the real registry and shell:

- exactly twenty-two rows have unique `cover_path` values;
- every path exists and loads as `Texture2D`;
- all twelve console covers are 2:3;
- each handheld cover matches its row's 1:1, 9:16, or 16:9 label law;
- all twenty-two focusable cards are built in catalog order;
- card accessibility text preserves title, aspect, power, network, and lock
  state;
- square, portrait, landscape, and box covers use keep-aspect presentation;
- locked art remains visible but cannot launch;
- missing-art fallback remains visible and does not crash the shell; and
- a real card press preserves the existing handheld PLAY and console MATCH
  routes.

Run the existing registry, handheld catalog, console catalog, shell,
acquisition, device, lobby, save, input, license, and complete Game Deck suites
after the focused test.

Render and inspect at least these GPU frames:

1. the ten-cover Pocket Game Deck library with all three label ratios visible;
2. the twelve-cover Safehouse Console library with owned and locked states;
3. WASTE HEAP focused at the 720 x 600 minimum; and
4. RUST RUNNERS focused at 1280 x 720, then opened into MATCH.

Inspect every individual final and the complete contact sheet for title
accuracy, forbidden purple, real-world trademarks, accidental lettering,
watermarks, distorted subjects, clipped dominant silhouettes, and thumbnail
legibility.

## Completion gate

The cover library is complete only when all twenty-two original final assets
exist in `res://`, every catalog path and dimension passes, the real library
shows all covers with controller/keyboard/mouse access, all regressions pass,
the four GPU frames and contact sheet are visually accepted, temporary key art
is removed, the worktree is clean, and the result is committed on the isolated
Game Deck branch.
